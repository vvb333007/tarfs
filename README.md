## TARFS — Read-only mmap-oriented filesystem over TAR images for firmware assets and runtime resources

### 1. Introduction

TARFS is a lightweight read-only filesystem designed specifically for embedded systems.

The filesystem image is a standard TAR archive generated by the GNU `tar` utility.

Instead of implementing a full writable filesystem, TARFS provides direct zero-copy access to standard 
POSIX TAR archives stored in memory-mapped Flash. Files of any size can be accessed with virtually no 
RAM overhead while remaining compatible with familiar POSIX interfaces and standard TAR tools.

Most files included in firmware never change after the image has been built. HTML pages, CSS, 
JavaScript, images, certificates, fonts, dictionaries, lookup tables, ML models, and similar 
resources are typically generated during the build process and are only read at runtime. Using 
a fully writable filesystem to store such data is often unnecessary.

---

### 2. Why another filesystem?

1. **Efficient storage of immutable assets on memory-constrained systems with a large number of files.**

   Typical examples include firmware/UI assets, icons, embedded HTTP server content, fonts, certificates, and similar resources.

   TARFS is optimized for a large number of read-only files and fast lookup.

2. **A porting layer for software originally written for Linux.**

   Many applications expect a reasonably POSIX-compatible filesystem. For example, the PH7 PHP Engine for ESP32 uses `mmap()` to execute PHP scripts directly from files.

   Since TARFS implements `mmap()` as true zero-copy mapping, applications can work with files of virtually any size without consuming additional RAM.

3. **Improved robustness for field devices.**

   Even after filesystem corruption, TARFS can usually still mount the archive and recover access to the remaining valid files.

4. **Per-file integrity verification.**

   Every filesystem object can be individually verified using embedded CRC checksums.

---

### 3. Features

#### 3.1 Nothing revolutionary

TARFS does not introduce a new filesystem format. It simply provides a reasonably POSIX-compatible filesystem interface (POSIX in terms of the API) over a standard TAR archive.

It is written in C, requires only a single platform-dependent source file, and is intended for microcontrollers capable of memory-mapping Flash.

Potential targets include:

* STM32
* ESP32
* RP2040
* SAMD21/SAMD51
* nRF52
* RA4M1 (Arduino UNO R4)
* LPC17xx/LPC55xx
* and many other MCUs that expose Flash through the processor address space.

The TAR archive may reside in external Flash, an internal Flash partition, or be linked directly into the firmware image using the linker or binary embedding mechanisms.

In particular on ESP32, you can use the `EMBED_FILES` mechanism (see ESP-IDF docs) to embed a TAR archive into the firmware 
image. The embedded archive can then be mounted using the `tarfs_mount_from_memory()` API.


---

#### 3.2 Memory requirements

TARFS is designed for memory-constrained microcontrollers.

The filesystem requires **24 bytes of RAM per file or directory**. A filesystem containing 1000 entries occupies slightly less than **24 KB** of RAM.

The number of active `mmap()` mappings is unlimited because `mmap()` does not allocate any resources. It simply returns a pointer into the mapped archive.

The filesystem index stores direct pointers to file names already present inside the TAR archive. No additional memory is allocated for file names.

During normal operation TARFS performs **no dynamic memory allocation at all**. Memory allocation only occurs during `mount()` and `unmount()`.

Because only two heap allocations are performed during mounting, heap fragmentation is practically eliminated.

---

#### 3.3 Better POSIX compatibility than existing ESP-IDF filesystems

TARFS supports true zero-copy `mmap()`. Files become directly accessible through memory without copying them into RAM.

List of supported features include:

* `readlink()`
* `fdopendir()`
* `mmap()`
* symbolic links, hard links and Windows directory junctions (when present in TAR archives created on Windows)
* UTF-8 file, directory, and link names
* opening directories with `open()`


---

#### 3.4 Per-file integrity verification (CRC64/ECMA-182)

Each TAR record may contain a CRC64/ECMA-182 checksum without affecting TAR compatibility.

Checksums are stored inside unused header fields, allowing the archive to remain fully readable by standard TAR utilities.

The accompanying utility `tarsum.c` computes CRC64 checksums for every archive entry with non-zero data size (including non-file records) and stores them in the unused header space.

---

#### 3.5 Corruption tolerance

Unlike conventional filesystems, TARFS has no central superblock.

As a result, damaging a few archive records usually does not render the entire filesystem unusable.

Corrupted TAR entries are skipped during mounting while valid records remain accessible.

TARFS is also tolerant of forgotten `munmap()` calls. Since `mmap()` does not allocate memory or other runtime resources, failing to call `munmap()` cannot cause memory leaks or resource exhaustion.

---

#### 3.6 Performance and predictability

Raw `read()`, `pread()` and `fread()` speeds are around 26MiB/sec (tight loop, 1KiB reads to the buffer 
from a 4MiB file on ESP32-S3 240MHz, Flash: 80MHz quad).


With the exception of `mount()` and `unmount()`, TARFS does not use synchronization primitives.

The entire runtime data path is lock-free. Some race conditions are possible, but they are limited to well-defined situations and are documented.

Inodes are located using binary search on the filename hash.

For directory iteration (`readdir()` and related functions), the inode table is sorted both by hash and lexicographically.

| Operation                                      | Complexity     | Notes                                         |
| ---------------------------------------------- | -------------- | --------------------------------------------- |
| `mount()`                                      | `O(3N) + O(k)` | Three archive passes plus link resolution     |
| `open()`, `opendir()`                          | `O(log N)`     | Binary search                                 |
| `readlink()`                                   | `O(k)`         | *k* = total number of symbolic and hard links |
| `read()`, `mmap()`, `close()`, `lseek()`, etc. | `O(1)`         | Direct access through precomputed pointers    |
| `readdir()`, `telldir()`, etc.                 | `O(1)`         | Sequential traversal using precomputed links  |
